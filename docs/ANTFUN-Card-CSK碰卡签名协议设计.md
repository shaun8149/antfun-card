# CSK 碰卡签名协议设计(纸面 · 待 B1 真机验证后落码)

> 2026-07 · 依据《安全威胁模型》§0.2/§1.1/§2.1/§2.4 与《业务构想》场景 ①②⑥。
> **状态**:协议层已定;**代码待 B1(真机 NFC,尤其 iOS)结论**——因为拓扑决定了哪条流用活签名、哪条用静态令牌。

---

## 一、CSK 是什么、放在哪

**CSK(名片签名密钥)= 卡的身份签名密钥,面向"他人"证明"真卡真人在场"。** 与另两把密钥严格隔离:

| 密钥 | 面向 | 用途 | 泄漏后果 |
|---|---|---|---|
| 资产私钥 | 链 | 签交易 | 丢钱 |
| DAK 设备认证 | 平台 | 证明"我方真卡"(激活验真、clone 接收方认证) | 供应链 |
| **CSK 名片签名** | **他人** | **碰卡加好友、见面证明、碰卡绑定、反女巫最高权重边** | **仅社交层** |

**落点**:独立的**名片 applet**(与钱包 applet 密钥域隔离,CSK 签名**不能触发转账签名**)。**模板 = Keycard 的 CashApplet**——它已经是"独立密钥对 + 对一个 hash 签名 + 带公开数据文件"的形态,几乎就是 CSK 的骨架。

---

## 二、两种碰卡拓扑(关键:别混)

碰卡在业务里其实是**两条不同的流**,只有一条用活的 CSK 挑战-响应:

### 流 A —— 双方都有 App(见面证明 / 反女巫最高权重边)
两个已装 App 的用户碰卡 → 卡做**活挑战-响应签名**(不可伪造)。这是 **CSK 的主场**,产出"线下认证边"(反女巫最高权重)、见面徽章。
- Android:reader/HCE 模式可直接 APDU。
- iOS:App 前台 + CoreNFC + Info.plist 声明 AID,可做 APDU 活签名。**(B1 要确认能到哪步)**

### 流 B —— 冷启动邀请(新用户,没装 App)
老用户 A 的卡被新用户 B 的手机碰一下 → B 手机**后台读 A 卡的 NDEF**(静态)→ 拉起 App Store/App → deferred deep-link 把 A 的邀请带进来。
- **这条流 iOS 上做不了活签名**(B 没装 App、只能读 NDEF),静态 NDEF 又可被一元贴纸复制。
- **所以冷启动绑定不靠 CSK 活签名,而靠"库存链归因"**(业务构想:归因主装置是库存链、碰卡是兜底)+ 静态签名 deep-link 令牌(仅 UX/兜底,权重最低)。

> **设计定论(解掉之前那个张力)**:"关系先于 App"(流 B)与"不可伪造"(流 A)是**两条流**,不是一件事。不可伪造的 CSK 只用于**双方在场、都有 App** 的见面证明与反女巫边;冷启动邀请由库存链authoritative,静态令牌只兜底。

## 三、CSK 挑战-响应协议(流 A · APDU 级)

**名片 applet 命令**(独立 AID):

```
(perso)  CSK 密钥对卡内生成;CSK 公钥导出 → 随 serial 登记入平台(徽章/边验签只认注册公钥)
INS_CSK_TAP_SIGN:
  App → 卡 : nonce(16B 服务端随机) ‖ ts(时间戳) ‖ ctx(场景/对端ID, 变长)
  卡      : counter += 1  (卡内持久单调计数器)
            sig = ECDSA_CSK( SHA256( "ANTFUN-TAP-v1" ‖ nonce ‖ ts ‖ ctx ‖ serial ‖ counter ) )
  卡 → App: sig ‖ serial ‖ counter
  App → 服务端: 上报 → 用 serial 查 CSK 公钥 → 验签 + nonce 5 分钟查重 + counter 单调校验
```

要点:
- **卡对完整挑战哈希签名**(不是签 App 给的裸 hash),并把 **域分隔前缀 "ANTFUN-TAP-v1" + serial + counter** 一起纳入——这样一条 CSK 签名**语义唯一**,不会被挪作他用;且和资产私钥不同域,CSK 签名对钱包无意义。
- **counter 卡内持久单调**,每次签名 +1,防重放的纵深防御(nonce 查重是主防)。

### 见面证明(高价值)—— 双向签名
"排队碰老板的卡领徽章":关键人物卡的 CSK 对 `本人在场` 背书。防滥用:
- 关键人物卡签发设**时间窗 + 场景 ID**(仅会场当日有效);
- 高价值徽章要求**双向签名**(双方卡都在场,各自 CSK 对对方 nonce 签一次)。

## 四、安全性质 ↔ 威胁映射

| 威胁 | 对策(本协议) |
|---|---|
| 静态卡克隆 / 签名重放 | 挑战-响应:克隆卡无 CSK 私钥,签不出;nonce 一次性 + counter 单调 |
| NFC 中继(relay) | ts 时间窗 ≤10s + ctx 绑定;见面徽章高价值场景加 GPS/地理围栏一致 |
| 借卡代签见面徽章 | 关键人物卡时间窗 + 场景 ID;高价值双向签名(双方在场) |
| 伪造"老板卡"骗签 | CSK 公钥在平台注册,徽章验签**只认注册公钥**,山寨卡签名不被承认 |
| CSK 泄漏 | 只影响社交层;资产层(不同密钥域)不受影响 |
| 名片签名触发转账 | 独立 applet + 独立密钥 + 域分隔前缀,结构上不可能 |

## 五、applet 设计:借什么 / 净新增

**借(CashApplet 现成)**:独立 CSK 密钥对(装机/perso 生成)、`sign` 对 hash 签名并回 pubkey+sig、公开数据文件(放 serial)。

**净新增(小)**:
1. 把 `sign` 从"签裸 hash"改成"卡内 SHA256(域前缀 ‖ nonce ‖ ts ‖ ctx ‖ serial ‖ counter)"再签;
2. 加**持久单调 counter**(每签 +1);
3. 响应带 serial + counter;
4. NDEF 记录:流 B 的静态签名 deep-link 令牌(拉起 App + 兜底邀请)——注意它的权重最低、可复制,只作 UX。

**可仿真器 TDD 的部分**:INS_CSK_TAP_SIGN 的签名/counter/serial 逻辑,和 clone 一样能在 jcardsim 上先写先测。**不依赖硬件。**

## 六、必须等 B1 才能定的(落码前的开放项)

1. **iOS 活签名到底能到哪步**:App 前台 CoreNFC APDU 会话能否稳定完成一次挑战-响应?延迟?→ 决定流 A 在 iOS 的可行性与 UX。
2. **iOS 后台 NDEF 拉起**的实际行为(URL scheme / universal link / App Store 跳转链路)→ 决定流 B 令牌怎么放、deep-link 怎么带。
3. **ctx/nonce/ts 的确切字节格式与长度上限**(受 APDU/NDEF 容量约束)。
4. **counter 位宽**(碰卡频次 × 寿命)与溢出策略。
5. 双向见面证明的**双卡编排**在真机上怎么走(一次会话两次碰,还是两次会话)。
