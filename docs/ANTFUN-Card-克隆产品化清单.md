# 卡组克隆 —— 阶段 2 产品化清单

> 来源:clone-spike 分支完成后的最终整分支审查(2026-07)。spike 在仿真器上证明了协议正确、核心安全属性(卡内认证接收方并绑定 ECDH 对端)成立、无明文密钥泄漏。以下是从 spike 走向产品必须处理的项。

## 阻塞项(必须在产品化前修)

> **状态更新(2026-07,A1 完成)**:①PIN 门控 ✅ **已在仿真器实现**——CLONE_EXPORT/IMPORT 均需 `pin.isValidated()`,并有"注释掉门控则测试挂"的实证测试;②getS ✅ **已做防御性右对齐修复**(零填 32 槽 + 右对齐),但**真卡上 getS 是否真会返回 <32 仍需 B2 实测确认**。其余项仍待产品化处理。

1. **EXPORT/IMPORT 加 PIN / 安全通道门控(最高优先)。** ✅ 仿真器已做(A1)。
   当前 `INS_CLONE` 走明文分发、不需 PIN。证书只证明"接收方是某张 CA 认证的真卡",不证明是"用户本人的卡"。残余威胁:**持有 A 卡 + 任意一张真卡的攻击者即可导出主密钥**(买一张合法卡就有合法 DAK 证书)。修法:EXPORT 与 IMPORT 均要求 `pin.isValidated()`;克隆作为独立高敏操作、单独确认/重输 PIN(见《结论详解》§八)。这也与"接收方绑到 owner/账户"的待议项 [[ANTFUN-Card-待讨论项]] D-1 相关。

2. **`masterPrivate.getS()` 返回长度不能假设恒为 32。**
   某些卡对有前导零的私钥不左补齐,`getS` 可能写 <32 字节 → 链码错位 → B 装载错误密钥(~1/256,jcardsim 恒返回 32 故测试不触发)。修法:取返回长度,非 32 则拒绝或左补齐进种子缓冲;perso 时也审计卡侧行为。

## 应做项

3. **IMPORT 侧对 `e_A_pub` 做曲线点校验。** 导入方对攻击者可控的 `e_A_pub` 做 ECDH,无证书。`ALG_EC_SVDP_DH_PLAIN` 通常会校验点,但要在**目标 SE 上实测确认**抗无效曲线攻击,不能假设。
4. **CA 公钥改为 perso 注入,删除 `CLONE_P1_SET_CA` 命令。**
5. **nonce 两个边界**:(a) `lastCloneNonce` 全零初值 → 首个全零 nonce 会误判重放,改用 hasLastNonce 标志或哨兵初值;(b) nonce 在下游加密步骤(可能抛异常)**之前**就记录 → 失败后同 nonce 的正当重试被误拒,改为**完全成功后**才记录。
6. **nonce 语义澄清**:只记"上一个" nonce 是弱新鲜性,且因每次导出临时密钥都是全新随机,nonce 的真实抗重放价值有限——它更像是 B 卡选定的 HKDF salt。产品化时明确 nonce 到底买了什么,并写进文档。
7. **成功响应给一个有类型的小 body**,让客户端不必 re-select 才能读 key UID。

## 建议补的测试

8. **真正的两卡端到端**:把一次真实 EXPORT 的输出直接喂给另一个 applet 实例的 IMPORT(目前两半分别证明)。这能正面覆盖 `getS` 长度/对齐这一类 bug。
9. **导入侧负例**:用随机/攻击者 `e_A_pub` + 为不同密钥算的 tag 导入,必须失败——把导入侧信任边界写成测试。

## spike 里已有的、刻意的简化(产品化时替换,非缺陷)

- AES-CBC + HMAC-SHA256(encrypt-then-MAC)代替单一 AEAD(目标卡无 AES-GCM)。
- 零 IV(合理:每次导出 AES 密钥因临时 ECDH 而唯一)。
- `CLONE_P1_SET_CA` 命令代替 perso 注入 CA。
- 无 PIN 门控(见阻塞项 1)。
- 成功返回裸 0x9000(见应做项 7)。

## 已验证成立(阶段 1 的地基)

- 卡内验 DAK 证书,且**验签的公钥 = ECDH 的对端**(无 TOCTOU)——整套设计的核心安全属性。
- 导入密码学上是导出的逆;先验 tag 再解密;篡改被拒。
- 任何克隆路径都不以明文返回主密钥/共享密钥/派生密钥。
- 测试为真实行为(BC 扮演对端卡,实算 ECDH/HKDF/AES/HMAC,比对 key UID)。
