# ANTFUN Card 安全威胁模型

> 技术规格 · 两轮红队对抗 · 残余风险 · 机密 · 2026-07
> **修订说明**:§1.5 已于 2026-07 改写(原「完成即熔断」→「默认可迁移 + 可选封印」,并把接收方校验从 App 侧移入卡内)。§2.1、§3 连带同步。改写背景见 `ANTFUN-Card-威胁模型-1.5改写稿.md` 与 `ANTFUN-Card-结论详解.md`。

**一票否决红线:私钥与助记词永不出厂、永不出芯片**——用户激活时在卡内自生成,任何「预置种子」方案一律不采用。

**信任边界**:仅 EAL6+ 安全芯片内部可信;卡厂、代工、物流、运营人员、用户手机、网络一律默认不可信。本文档给出可实现的技术规格,非原则宣示。

## 0　系统构成与密钥层级

### 0.1　一卡双应用(JavaCard)

**钱包 applet**(基于 Status Keycard 开源实现):HD 主密钥在卡内生成,secp256k1 签名(BSC / EVM),PIN 保护 + 错误计数锁死(建议 3 次 PIN、5 次 PUK)。私钥无任何导出指令。默认不向用户展示助记词——备份由卡组克隆完成(见 §1.5)。

**单芯片架构(关键设计理由)**:钱包与名片两个 applet 跑在同一颗 EAL6+ 安全芯片内,不存在通用 MCU 与安全芯片之间的通信总线。2023 年 OneKey Mini 的严重漏洞正是出在这条总线上(在 CPU 与安全单元之间插入 FPGA、诱导进入工厂模式导出助记词)——单芯片方案对这一整类攻击结构性免疫。

**名片 applet**(自研):NDEF 记录(碰卡拉起 App)+ 挑战-响应签名接口(见 §2.1)。与钱包 applet 密钥域隔离,名片签名**不能**触发转账签名。

### 0.2　四类密钥(职责严格分离)

| 密钥 | 生成位置 / 时机 | 用途与保管 |
|---|---|---|
| 用户资产私钥 | 卡内,用户**激活时**生成 | 签交易;永不出芯片(卡组克隆例外:仅以目标卡公钥加密的一次性通道内复制,见 §1.5);平台不可见、不可恢复 |
| 设备认证密钥 DAK | 我方 HSM 生成密钥对 → perso 时注入卡内 | 证明「这是我方出厂的真卡」;公钥入库,卡厂只见密文注入包 |
| 名片签名密钥 CSK | 卡内生成(perso 触发) | 签名 tap(见面证明 / 碰卡绑定);公钥随序列号登记 |
| 固件签名密钥 FSK | HSM,多人仪式 | 签发 applet 版本;任何单人不可签发;离线保管 |

DAK 与 CSK 分离的原因:DAK 证明真伪(面向平台),CSK 证明在场(面向他人)。CSK 若泄漏只影响社交层,资产层不受影响。

## 1　核心协议规格

### 1.1　挑战-响应碰卡(防克隆、防重放)

静态 NDEF 卡可被一元钱的贴纸复制,因此**不采用静态 URL**。每次碰卡执行:

```
App  →  卡 :  nonce(16B 随机) ‖ ts(时间戳) ‖ ctx(场景ID)
卡   →  App:  sig = ECDSA_CSK( SHA256(nonce‖ts‖ctx) ) ‖ serial ‖ counter
App  →  服务端: 上报 sig / serial / counter → 验签 + nonce 查重 + 计数器单调递增校验
```

- **防克隆**:克隆卡无 CSK 私钥,无法应答挑战。
- **防重放**:nonce 一次性(服务端 5 分钟内查重)+ 卡内计数器单调递增(回退即判伪)。
- **防中继(relay)**:ts 时间窗 ≤10 秒;见面徽章等高价值场景额外校验双方 GPS 一致性与会场地理围栏。

### 1.2　激活与真伪验证

```
① 用户碰卡 → App 读 serial,卡用 DAK 对服务端 challenge 签名
② 服务端查 serial 对应 DAK 公钥 → 验签通过 = 真卡;校验 serial 状态 = 未激活
③ 卡内生成主密钥(熵不出芯片)→ 卡组克隆至备份卡(§1.5)→ 卡返回主公钥
④ 服务端登记:serial → 用户 ANT 号;按库存链写入邀请关系;serial 置为已激活(一次性)
```

- **传输安全**:App 与服务端证书固定(cert pinning);③ 的熵生成全程在卡内,服务端只收公钥。
- **一卡一次激活**:二手卡 / 回收卡再激活被拒。

### 1.3　归因绑定(库存链优先)

序列号仅作索引,**绑定动作必须附卡内 CSK 签名**——即使序列号被预知或猜测,无卡不能绑定。库存转移(团队长→小队长)需双方 App 签名确认并留痕;中性库存(官网 / 公共售卡处)走碰卡兜底绑定。

**边权重**:线下签名碰卡边 > 库存链边 > 线上自绑边;大额激励与金 V 审核只认线下认证边。

### 1.4　perso(个人化)流程与卡厂隔离

```
我方 HSM: 生成 DAK 密钥对 → 用卡厂 KTK 加密成注入包(卡厂不见明文)
卡厂安全车间: GP 装载双 applet → 注入 DAK 包 → 写 serial → 打印卡面 → 防拆封装
出厂状态: 未激活、无种子、无用户数据;DAK 公钥与 serial 回传我方入库
```

- **卡厂能做的**:装载我方审计过的 applet、注入密文包。
- **卡厂不能做的**:读取任何私钥、写入种子、伪造 DAK(无 HSM 私钥)。批次全程可追溯,抽检验证固件哈希。

### 1.5　卡组克隆备份(取代助记词的核心设计)【2026-07 改写】

**产品目标**:让用户不再需要手动备份助记词;并支持在 NFC 消磁 / 卡片丢失后,把私钥迁移到一套全新卡组。

**形态**:一套 N 张卡(建议 1 主 + 2 备),互为完整副本;丢失任意一张,用另一张继续。主密钥以 512-bit 种子形态在卡间以密文复制,助记词从不存在。

**安全收益是决定性的**:全行业针对用户的终极攻击都是「请输入你的 24 个助记词」——用户手里根本没有助记词,这个攻击面直接消失。

#### 克隆协议(A=源卡,B=目标卡;既可同组备份,也可跨组迁移到新卡)

```
① B 卡输出:自身 DAK 证书(authentikey 公钥 + 我方 CA 签名)+ 卡内新生成 nonce_B
② App 转发①给 A(仅搬运,因有 CA 签名而不可篡改)
③ A 卡在【卡内】校验,任一不过即拒:
     a. 用内置 CA 公钥验证 B 的 DAK 证书签名 → 确认 B 是「我方出厂真卡」(非任意野公钥)
     b. 校验 nonce_B 新鲜 → 防重放
     + PIN 已验证
④ A 卡以【临时】ECDH(临时私钥 e_A × B 的 authentikey 公钥)导出共享密钥;
     KDF = HKDF-SHA256(ECDH ‖ nonce_B);AEAD(AES-CCM,密钥由 HKDF 派生)加密 512-bit 主种子;输出 密文 ‖ e_A 公钥
⑤ App 转发密文给 B;B 卡以 authentikey 私钥 + e_A 公钥 复算密钥、校验 nonce_B、AEAD 解密 → 卡内装载
⑥(可选·封印模式)用户若选「备份完成、永不再迁移」→ 置 OTP 熔断位 → 该卡克隆指令永久不可用
```

#### 关键安全性质

- **卡厂无法克隆**:套内联结靠各卡卡内自生成的 authentikey 公钥(公开信息);卡厂不持有任何可解密的私钥。DAK 证书由我方 HSM 的 CA 签发,卡厂只注入密文包。
- **接收方认证在卡内,不在 App**:第 ③a 步由 A 卡用内置 CA 公钥验证 B 的证书。木马 App 无法把主种子骗给一个野公钥——野公钥没有我方 CA 签名,A 卡直接拒绝。攻击者若想当合法目标,只能用一张自己购买的真卡,且导入后须物理持卡方能动用 → 攻击退化为「需实体卡 + 劫持一次克隆时刻」的可接受档。
- **临时 ECDH + nonce**:每次会话密钥不同,捕获的旧密文不可重放。
- **PIN 强制**:卡组模式下 PIN 为必选项;PIN 错误计数锁死;克隆作为独立高敏操作,要求单独确认 / 重新输入 PIN,不复用普通签名会话的 PIN 状态。

#### 默认可迁移 vs 可选封印

- **默认:克隆通道长期可用**(PIN + 卡内 DAK 认证门控)。合法目标是任何一张我方真卡,支持「源卡 → 全新目标卡」迁移,解决 NFC 消磁 / 丢卡后换卡刚需。
- **可选:封印模式**。面向「已备份完、要极致安全、永不再迁移」的高阶用户,自选置 OTP 熔断位(硬件一次性、独立于 factory-reset、不可逆)。

#### 代价与告知义务

- 全套卡同时丢失 = 资产永久不可恢复。产品必须强制告知并引导**分地点保管**。
- **放宽熔断带来的新残余风险**:克隆通道长期存活,意味着一次 PIN 已验证的会话中,恶意 App 可能静默发起一次「克隆到攻击者自购真卡」。缓解:①克隆需单独确认 / 重输 PIN;②主种子强制安全导出、禁明文路径;③克隆次数上限 / 冷却 + 告警;④高价值用户引导启用封印模式。面向高阶用户可提供 App 端加密备份作为第三条路径(可选,非默认)。
- **待议**:接收方是否收紧到「同 owner/账户」(而非「任何真卡」),见 `ANTFUN-Card-待讨论项.md` D-1。

## 3　复盘与红线

**已结构性关闭**:供应链植入种子、静态克隆、重放、库存劫持、孤儿认领、现场截胡、二手转售——由「种子不出厂 + 挑战响应 + CSK 签名绑定 + 一次激活」四条设计覆盖。克隆通道要求**卡内认证接收方**,野公钥被拒;跨组迁移为设计内允许行为(不再以「完成即熔断」为默认前提)。

**长期压制、无法根除**:NFC relay、线上女巫、钓鱼与恶意软件、托管信任成本、借卡代签、物理胁迫、PIN 会话内静默克隆到攻击者真卡。共性对策是**分层 + 权重 + 人工 + 教育**,不是单点技术。

**不可谈判红线**:种子永不出厂出芯片(克隆仅以目标卡公钥加密的密文形态、且经卡内 DAK 认证);固件开源 + 第三方审计 + 哈希公示;EAL6+ 芯片;FSK 由 HSM 多人仪式保管;孤儿不可事后认领;托管资金可审计;客户住址等 PII 最小化收集;见面徽章与榜单默认不公开可关联线下身份的信息。

**分层定位是最大的安全杠杆**:卡片款 = 随身小额 + 身份;大额资产走旗舰 / App 本体——把 relay、胁迫、恶意软件、遗失的最坏后果在定位层限额,比任何单点技术都有效。

## 4　行业事故复盘(Ledger / Trezor / OneKey)

**规律先行**:三家的设备核心从未被远程攻破,但用户持续受损——教训重心已从「设备会不会被破解」迁移到「厂商会不会泄露你是谁」。

| 公司 | 重大事件 | 应对与教训 |
|---|---|---|
| Ledger | 2020 电商库泄露约 27 万客户姓名 / 住址 / 电话;2021 按地址邮寄改装假设备;2023.05 Ledger Recover 引爆信任危机;2023.12 Connect Kit 供应链投毒(前员工被钓鱼 → npm 库植入盗币脚本,损失约 $60 万);2025.01 联合创始人遭绑架勒索;2025.04 实体信件二维码钓鱼;2026.01 第三方支付商 Global-e 泄露订单 | Connect Kit 数小时修复 + 赔付 + npm 强制 2FA 是响应标杆;但泄露的住址无法收回,因果链一路走到人身安全。教训:①最大脆弱点是客户数据不是设备;②产品公告也能是安全事故;③软件供应链与硬件同等致命 |
| Trezor | 2020 Kraken 电压毛刺物理提取种子(无安全芯片);2022 Mailchimp 泄露引发钓鱼;2024 客服工单系统泄露约 6.6 万联系方式;2025.03 竞争对手白帽披露 Safe 3 毛刺漏洞;2026.02 实体信件钓鱼 | 承认老硬件无法修复、建议 passphrase,下一代改用 EAL6+ 安全芯片(2025 Safe 7 含后量子与开源芯片)。教训:①无安全芯片 = 物理可提取;②三次泄露全部来自第三方;③竞争对手会公开审计你,要提前有披露政策 |
| OneKey | 2023.02 Unciphered 公布 Mini 严重漏洞:在 CPU 与安全单元之间插入 FPGA、诱导进入工厂模式导出助记词(需拆机 + 实验室设备);2025.10 Milk Sad 随机数事件主动澄清不受影响 | 当日发文说明已修复、无用户受影响、讲清攻击门槛,并向白帽支付赏金致谢。技术教训:破绽在 CPU–SE 总线而非安全芯片本身——我们的单芯片架构对此免疫(0.1) |
