## 一、为什么"卡组克隆"是整个产品的技术命门

我们的安全叙事只有一句话立得住才成立:**用户手里没有助记词,所以钓鱼无物可骗**。但"没有助记词"必然带来一个问题——**没有助记词,拿什么备份?** 答案是卡组克隆:一套 N 张卡互为副本,主密钥以密文在卡之间复制,复制通道用后即焚。

这意味着:**卡组克隆是我们唯一的备份机制,它一旦不安全,整个"无助记词"的地基就塌了**。所以它必须比助记词更安全,而不是仅仅"不一样"。这也是为什么本轮把最多的对抗火力压在它身上。

威胁模型的硬约束(来自《安全威胁模型》):**除 EAL6+ 芯片内部,一切默认不可信——包括用户的手机**。安卓手机可能带木马。克隆过程由手机 App 编排(手机是 NFC 桥,卡之间不能直接通信),所以**必须假设编排克隆的那台手机是敌人**。这一条,直接判了两个现成开源实现的死刑。

## 二、两个现成实现是怎么死的(攻防全过程)

### 2.1 Keycard 的 XOR 对称复制 —— 手机能重建密钥

**它怎么工作**(源码 `status-keycard@1b71679^:KeycardApplet.java:1105-1215`):

```
① START  : 手机 → 各卡下发一个初始 32B 值,作为 duplicationEncKey 的种子;告知参与方数量 N
② ADD_ENTROPY(重复 N 次): 每张卡/客户端各生成 32B 熵,经手机中转,
            接收卡把它 XOR 进自己的 duplicationEncKey
            → 所有卡最终得到同一个 AES-256 密钥 = 初始值 ⊕ 熵₁ ⊕ 熵₂ ⊕ …
③ EXPORT(源卡 A): 用该 AES 密钥 AES-256-CBC 加密"主私钥+链码",输出密文
④ IMPORT(目标卡 B): 用同一 AES 密钥解密,loadKeyPair 装载
```

**攻击(木马手机)**:第 ② 步里,每一份熵都要**经过手机**从一张卡送到另一张卡。卡与手机之间虽有安全通道,但**安全通道的另一端就是手机本身**——手机能看到每一份熵的明文。于是手机可以:

```
恶意手机 = 初始值(它自己下发的) ⊕ 它转发过的每一份熵
        = duplicationEncKey   ← 手机完整重建了 AES 密钥
第 ③ 步 A 卡吐出的密文,手机直接用这个密钥解密 → 拿到主私钥明文
```

**结论**:XOR 对称模型把密钥的所有材料都过了一遍手机的手,**编排者天然掌握密钥**。在"手机可信"的假设下没问题(Keycard 当年就是这么设计的),但在我们"手机不可信"的红线下,**直接照抄 = 把主私钥送给木马**。Keycard 后来用 BIP39 助记词替换了这个特性——而我们因为要 no-mnemonic,反而要把它复活,但**必须换掉密钥协商方式**。

### 2.2 SeedKeeper 的静态 ECDH 导出 —— 手机能把密钥骗到攻击者卡上

**它怎么工作**(`Seedkeeper-Applet:SeedKeeper.java:2249-2257` 等):

```
共享密钥 K = ECDH(A的authentikey私钥, B的authentikey公钥)
① 手机先读出 B 卡的 authentikey 公钥,把它当作一个普通 PUBKEY 对象导入 A 卡
② A 卡:K = ECDH(自己私钥, 手机给的那个"B公钥");
        AES 密钥 = HMAC-SHA1(K.x, "seckey");加密主密钥,encrypt-then-MAC(HMAC-SHA1)
③ 手机把密文送到 B;B 卡:同样 K,解密装载
```

比 XOR 好在:密钥是 A、B 卡的私钥参与算出来的,**手机看不到 K**(手机没有任何一张卡的 authentikey 私钥)。看似解决了 2.1 的问题。**但有两个致命洞**:

**洞一——目标公钥不被卡内验证(致命)**。第 ① 步里,"B 卡的公钥"是**手机口头声称**的,A 卡只检查它"长度 65、类型是 PUBKEY"(`:2230-2246`),**从不验证它真的属于一张同组的真卡**。攻击:

```
木马手机不给 A 卡真的 B 公钥,而是给"攻击者自己的公钥 PK_evil";
A 卡照算 K' = ECDH(A私钥, PK_evil),把主密钥加密给攻击者;
攻击者用自己的私钥解密 → 拿到主私钥
```

SeedKeeper 其实**预留了**认证接收方的字段(`PUBKEY_AUTHENTICATED=0x71`、`EXPORT_AUTHENTICATED=0x03`),但代码里**标着 RFU、根本没实现**(`:340,:361`)。也就是说,防止"导给攻击者"这件最核心的事,SeedKeeper 完全交给手机软件去把关——而手机正是敌人。

**洞二——静态-静态 ECDH,可重放**。K 对同一对卡永远不变,导出信封无新鲜性、无 nonce,`importSecret` 也不记 counter,所以**同一个密文可以被无限次重放导入**。加上 `resetToFactory` 还会把卡恢复到可重新克隆的状态(`:1117-1142`),等于熔断也没有。

### 2.3 收敛点:两者同一个病根

| | Keycard XOR | SeedKeeper 静态 ECDH |
|---|---|---|
| 木马手机怎么拿到主密钥 | 重建对称密钥直接解密 | 替换目标公钥,把密钥骗到攻击者卡 |
| **共同根因** | **卡内不认证接收方** | **卡内不认证接收方** |

一个把密钥材料交给手机,一个把"接收方是谁"交给手机——**都把信任放在了那台不可信的手机上**。所以我们真正要新写的,不是"加密传输"(那有骨架可抄),而是**让卡自己确认"对面是不是我的合法同组卡",且这个确认不经手机之手**。

## 三、我们的克隆协议应该长什么样(合成设计)

三份仓库正好各出一块拼图,拼起来就能堵住上面所有洞。下面是激活期一次性克隆的完整设想(概念级,非最终规格):

> **本节已按用户反馈修订**:去掉「同 set-id 绑定」和「激活后永久熔断」。原因见下节末尾的「设计取舍」——克隆必须支持跨组迁移(NFC 消磁/丢卡后迁到全新卡组),所以合法目标是**任何一张我们出厂的真卡**,而不是「同一套卡」;永久熔断降为**可选封印模式**。

**前置(perso 出厂时)**:每张卡的 IdentApplet 里注入一张 **DAK 设备证书**——内容是"本卡 authentikey 公钥 + 我们 CA 的签名"。**不再要求同 set-id**——每张真卡都是潜在的合法克隆目标。这一步 Keycard 的 IdentApplet 已经现成(`IdentApplet.java` + 钱包 applet 在 `KeycardApplet.java:218` 一进门就强制校验设备证书)。

**克隆流程(A=源卡,B=目标卡;既可同组备份,也可跨组迁移到新卡)**:

```
① B 卡输出:自己的 DAK 证书(authentikey 公钥 + CA 签名)+ 一个卡内新生成的 nonce_B
② 手机把①原样转发给 A 卡(手机只是搬运,改不了——因为有 CA 签名)
③ A 卡在【卡内】做两件事,任何一件不过就拒绝:
     a. 用内置的 CA 公钥验证 B 的 DAK 证书签名  → 确认 B 是「我们出厂的真卡」(而非野公钥)
     b. 记住/校验 nonce_B                        → 新鲜性,防重放
     + PIN 校验通过
④ A 卡用【临时】ECDH:临时私钥 e_A × B证书里的authentikey公钥 → 共享密钥;
     KDF = HKDF-SHA256(ECDH ‖ nonce_B);        ← 换掉 SeedKeeper 的静态ECDH+SHA1
     用 AEAD 加密「512-bit 主种子」;输出 密文 ‖ e_A公钥
     ※ AEAD 复用 Keycard 现成的 aesCcm*(AES-CCM),密钥由上面 HKDF 派生——
       不用 AES-GCM:目标级 EAL6+ 卡普遍无原生 GCM,Keycard 正是为此手搓了 CCM
⑤ 手机转发密文给 B;B 卡:用自己 authentikey 私钥 + e_A公钥 复算共享密钥,
     校验 nonce_B 是自己刚发的,AEAD 解密 → 卡内装载主种子
⑥(可选·封印模式)高阶用户若选「备份完成、永不再迁移」→ 置【OTP 熔断位】→ 该卡克隆指令永久不可用
```

**每个洞是怎么堵上的**:

| 攻击 | 对应防御 |
|---|---|
| 木马重建对称密钥(Keycard XOR 病) | 密钥来自临时 ECDH,私钥在卡内,手机拿不到 |
| 替换目标为**野公钥**、全程远程窃取(纯 SeedKeeper 病) | 第 ③a:卡内验 CA 签名,野公钥无我们签发的证书,过不了 → 攻击者被迫用**自己买的真卡**当目标、且导入后须**物理持卡**才能用 → 退化成「需实体卡 + 劫持克隆时刻」的可接受档 |
| 重放旧密文 | 第 ③b/⑤:nonce_B 每次新生成并进 KDF,旧密文换卡/换次都失效 |

**这里的净新增核心 = 卡内接收方 DAK 认证 + 临时 ECDH/nonce**,是审计中心。可复用:SeedKeeper 的信封骨架(升级 primitives)+ 其「任意点对点导出」正是跨组迁移所需、Keycard 的 IdentApplet 证书与 loadKeyPair。**注意**:去掉永久熔断后,克隆通道默认长期存活(PIN 门控),这引入的新残余风险(PIN 会话内被静默触发克隆等)在文末《放宽熔断后的新残余风险(第二轮对抗)》展开。

## 四、封印模式(可选)——熔断为什么不能照 Satodime 的密封/揭封抄

> **定位变了**:按用户反馈,永久熔断**不再是默认**,而是给「已备份完、要极致安全、永不再迁移」的高阶用户的**可选封印模式**。默认保持克隆通道可用(PIN 门控),以支持换卡迁移。下面讲的是:**一旦要做这个可选封印,怎么做才对**。

Satodime 看起来像"密封=锁、揭封=开",很容易被当成一次性熔断。但源码显示它是**可循环重置的环**:

```
STATE_UNINIT → (seal) → SEALED → (unseal) → UNSEALED → (reset) → STATE_UNINIT → …
```

`resetSatodimeKey`(`Satodime.java:1363`)会擦掉槽位、重新布设曲线参数,下一次 `seal` 生成**全新密钥**。它本来就是设计成**可重复使用的持票卡**——和我们要的"一次性、永不再开"正好相反。**若照抄,我们"已禁用"的克隆通道会被一次 reset 重新打开。**

Satodime 里**唯一真正不可逆**的,是另一处完全不同的机制——`personalizationDone`(`:309,724,1769`):一个持久布尔量,`setup()`/`lock_PKI()` 里置 `true`,**全代码没有任何地方把它置回 false**,置位后永久拒绝改证书。**这才是熔断范式**。

我们要在此基础上再硬化两点(因为它守的是高价值主密钥):

1. **用 OTP/fuse 位而不是普通 EEPROM 布尔**——普通布尔的"永久性"只等于那个 EEPROM 单元不被改写;OTP 是硬件层面一次性熔丝,写了物理上回不去。
2. **独立于 factory-reset**——SeedKeeper 的教训是它的 `resetToFactory` 会把卡恢复到可再克隆(它的 personalizationDone 虽不被 reset 清,但克隆通道本就没被它守)。我们的熔断位必须让 factory-reset **也清不掉、也绕不过**。

**设计取舍(为什么默认不熔断)**:原《安全威胁模型》1.5 写的是「完成即熔断」——好处是激活后密钥永不可再导出,坏处是**把换卡迁移一并焊死**。而 NFC 消磁、卡片丢失后用户换一套新卡是刚需,所以默认必须允许「源卡→全新目标卡」的克隆。代价是克隆通道长期存活,引入新残余风险(见文末《放宽熔断后的新残余风险》),用 **PIN 门控 + 卡内 DAK 认证 + 临时 ECDH** 把它压到「需实体卡 + 劫持克隆时刻」这一可接受档;想要 1.5 那种极致密封的用户,可自选启用本节的封印模式。**这是自觉的产品取舍:可替换性 > 密封性。**

## 五、ed25519 / Solana:两条路的真实代价

### 5.1 为什么通用 ECC 引擎做不了

BTC/ETH 用 secp256k1,是**短 Weierstrass 曲线**(`y²=x³+ax+b`),cofactor=1。JavaCard 的通用 ECC 引擎(`TYPE_EC_FP`)就是为这类曲线设计的。

Solana 用 ed25519,底层是 **Curve25519**——**Montgomery/twisted-Edwards 曲线,cofactor=8**。它**在数学形式上不是** `TYPE_EC_FP` 能表达的短 Weierstrass 曲线;而且 EdDSA 的签名机制(内部对消息做哈希、输出不是 DER 的 r/s)也和 ECDSA 不同。所以**不能"把 25519 当成又一条硬编码曲线塞进通用引擎"**——T2OpenPGP 的 `ECConstants.java` 里只有 6 条 Weierstrass 曲线,25519 走的是芯片里**另一条专有代码路径**,这一点被源码坐实。

### 5.2 两条能走的路,各自的真实代价

**路 A:原生 `ALG_EDDSA` 的 JavaCard 3.1 卡**
- 代码可移植、不涉 NDA,是最干净的路。
- **风险**:很多卡**声明**支持 3.1,但那个算法常量**未必真的实现**。必须用 **JCAlgTest / ECTester 上真卡实测**,签一笔真 ed25519 出来,才算数。"datasheet 写了"不作数。

**路 B:芯片厂专有 25519 库(Token2 走的路)**
- 真实存在、能量产,但那段代码是**芯片特定、NDA 锁定、不可移植**的固件("换一颗芯片就编译不了")。
- **对我们尤其相关**:文档偏好的国产 EAL6+(紫光同芯/华大)大概率落在这一类(顺带,Token2 用的 THD89 本身就是国产 SE)。也就是说 **SOL 在国产卡上,很可能要签 NDA、用芯片厂 SDK 写一段绑死该芯片的代码**,不是抄一段开源 JavaCard。

**软件实现(JCEd25519)已被否**:作者与底层库作者都加粗写"不可用于生产",威胁是**计时侧信道提取私钥**——而硬件钱包的全部价值就是抗物理攻击者,恰恰是这个威胁模型下最不能用。

### 5.3 由此得出的产品决断

**首发只上 secp256k1 系(BTC + 全部 EVM/BSC),SOL 延后**,与《业务构想》"首发定位 BSC/EVM"一致。SOL 的前提是:选到一张 JCAlgTest 实测原生 `ALG_EDDSA` 的 3.1 卡,或愿意为某国产芯片签 NDA 写专有代码。**不让 SOL 阻塞首发。**

## 六、钱包核心与其余模块(为什么是 Keycard,借了谁)

**钱包核心 = Keycard,已被 Satochip 对比确认**。决定性一条:Satochip **只能导入主机生成的种子,不能卡内生成**(`CardEdge.java:1654`),意味着种子必然在手机上存在过一次——对"密钥不出芯片"是硬伤;Keycard 的 `GENERATE_KEY` 卡内生成随机种子,主密钥从不经过手机。安全通道 Keycard 也更强(有配对密钥 + HKDF,Satochip 无配对、SHA-1)。

**从 Satochip 值得借的一个概念——2FA 金额门限协签**(`CardEdge.java:2326-2341`):卡内累加交易输出金额,超过预设 `2FA_limit` 就**拒绝签名,除非第二设备对交易哈希给出 HMAC 协签**。这是**无屏卡对抗恶意主机大额盗签的唯一机制**,Keycard 没有。局限:它只门限**金额**、不校验**目标地址**,且是 BTC 专用解析器。

**我们的分层对策**:卡片款定位小额、走盲签;**大额走带屏旗舰、屏上确认地址**。2FA 协签作为"若要做无屏高价值路径"的参考,非首发阻塞。

**红包卡**:借 Satodime 的 **commit-reveal 熵方案**(封印只给公钥承诺、揭封才给私钥+熵)——它让"送礼人无法预知或留存这把密钥"可被收礼人**密码学验证**,是可信红包的关键;再补"领取后自毁/清扫"。

**无助记词减法**:砍 `GENERATE_MNEMONIC`、`LOAD_KEY(SEED)`、`EXPORT_BIP85`、EIP-1581 私钥导出;`GENERATE_KEY`(卡内随机种子、全程无词)即无助记词主路径,卡组克隆做它的备份。

## 七、工程量、阻塞与顺序(汇总)

**白拿**:Keycard 钱包核心、签名(secp256k1+Schnorr)、PIN/PUK、安全通道、设备证书闸门、名片签名模板、NDEF、jcardsim 测试链。

**合成+新写(最高审计权重)**:接收方认证的 ECIES 克隆 + OTP 熔断。有三份参考托底,但安全关键胶水是真新写。

**未解阻塞(选型)**:所选 EAL6+ 卡的①算法集(`ALG_EC_SVDP_DH_PLAIN_XY`/`HMAC_SHA512` 等 Keycard 依赖)与②ed25519 是否原生,**都要 JCAlgTest/ECTester 上真卡实测**。这是唯一能让后续全部作废的单点。

**顺序**:
1. 芯片/卡选型 + 算法实测(先做,防作废)
2. jcardsim 仿真器 spike:跑通"卡内验 DAK 证书+set-id → 临时 ECDH+nonce → ECIES → OTP 熔断",不碰真卡
3. spike 通过 → 出正式《卡组克隆 applet 技术规格》→ 实卡开发

## 八、放宽熔断后的新残余风险(第二轮对抗)

用新逻辑重跑对抗时,SeedKeeper 的源码分析直接把放宽熔断的代价摆了出来。逐条记录:

**风险 1(本质性)——"任何真卡"= 任何攻击者可控的落点。** DAK 证书只能证明目标是"**我方出厂的真卡**",**证明不了它是"用户本人的卡"**。攻击者只要拥有**任意一张真卡**(自己买的/偷来的),就持有合法 DAK。而克隆的门槛只有 **PIN + 一个目标公钥**——所以**跨组迁移和跨组盗取,是同一个操作**。一旦 PIN 被胁迫/泄露,攻击者就能把主种子完整克隆到自己的真卡。这是放宽熔断换来的根本残余,无法用技术根除,只能压低。

**风险 2——无上限、无冷却 = 无限次克隆。** SeedKeeper 的导出计数器(`export_nb_*`)只用于日志、不作上限。我们必须自己加约束。

**风险 3——静态证书 ≠ 在场/持有证明。** 只验一张静态 DAK 证书,不能证明目标**当下真的持有**对应私钥(一个被复制的证书 blob 也能通过验证)。
- **澄清(修正我第二轮的初判)**:这一条**不会直接泄露种子**——因为第 ④ 步是把种子加密到目标公钥,只有物理持有对应私钥的那张卡能解密。攻击者拿一张"别人的静态证书"当目标,得到的是一段**他自己解不开**的密文。所以"证书重放"本身无害。
- 但它值得补一个**挑战-响应**(目标卡用 DAK 私钥现签一个新 nonce),确保目标**在场且活着**,消除"加密给一张不在场的卡"这类混淆。这是 nice-to-have,不是承重项。

**风险 4——明文导出路径必须强制关闭。** SeedKeeper 还留着 `SECRET_EXPORT_ALLOWED` 明文导出;对主种子这必须**强制禁用**,只允许安全、证书门控、临时 ECDH 的路径。

### 对策(默认档,把风险 1 压到"可接受")
- **克隆作为独立高敏操作**:要求**每次单独确认 / 重新输入 PIN**,不复用普通签名会话的 PIN 状态 → 堵"一次 PIN 会话内被木马静默发起克隆"。
- **主种子强制安全导出**、禁明文路径(风险 4)。
- **克隆次数上限 / 冷却 + 告警**(风险 2)。
- **目标挑战-响应**确保在场(风险 3)。

### 一个需要产品拍板的设计分叉:接收方绑到哪一层?

放宽熔断后,"合法目标"的范围有三档,是**安全 vs 灵活**的连续谱:

| 档 | 合法目标 | 跨组迁移 | 抗"导给攻击者真卡" | 代价 |
|---|---|---|---|---|
| 同 set-id(原设计) | 仅同一套卡 | ❌ 做不到 | 最强 | 焊死换卡刚需 |
| **同 owner/账户** | 绑定同一账户的任何真卡 | ✅ 可(新卡先绑到你账户) | **强**(攻击者的卡不是你账户的) | 需在卡内校验"owner-token",引入对账户/服务端签发的依赖 |
| 任何真卡(当前选择) | 任何我方真卡 | ✅ 最灵活 | 弱(仅"需实体卡+PIN") | 迁移=盗取同操作 |

当前默认是"**任何真卡**"。**"同 owner/账户"很可能是甜点**:既允许"换一套新卡迁移"(新卡先在你账户下激活拿到 owner-token),又能挡住"导给攻击者自己的真卡"(他的卡不在你账户下)。代价是克隆时目标要出示一个**账户绑定属性**(激活时由服务端签发、写入卡内),卡 A 在 DAK 验签之外再验这个属性匹配。**这一步要不要做,取决于你愿不愿意为它引入一点服务端/账户依赖——是个产品取舍,留待拍板。**
